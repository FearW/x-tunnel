## 后续值得做的优化（建议路线图）

下面这些是我认为对“速度、稳定性、可维护性”收益最高的方向，优先级按推荐顺序排列：

### P0（优先做，短周期见效）

1. **增加“自动测速选路”**
   - 对当前落地（直连/HTTP/SOCKS5/WG）做并行探测（延迟、抖动、下载速率）
   - 自动选择最优落地并可定时重评估
   - 价值：避免手工切换，实时拿到当前网络条件下最优路径

2. **增加“健康检查 + 自动恢复”守护**
   - 定时检查 `x-tunnel` / `cloudflared` / `wireproxy` 是否存活与可连通
   - 异常时自动重启单组件（而不是全量重启）
   - 价值：减少夜间断流和手工干预

3. **补一套结构化日志**
   - 将各组件输出统一落盘到 `~/.suoha/logs/`，按日期轮转
   - 提供“最近失败原因”快速查看入口
   - 价值：排障效率明显提升

### P1（中期优化，增强体验）

4. **多 WG 配置“策略组”**
   - 不只是手工热切换，而是支持：
     - 轮询 / 最低延迟优先 / 故障转移
   - 价值：WG 节点多时提升整体可用性

5. **按场景预设参数模板**
   - 例如：
     - 游戏低延迟模板
     - 大流量下载模板
     - 稳定常驻模板
   - 一键设置 `cf_protocol`、`ha_connections`、MTU、keepalive

6. **系统调优检测与回滚**
   - 启用 BBR+FQ 前先检测内核支持
   - 提供“恢复默认 sysctl”一键回滚
   - 价值：降低系统层调优风险

### P2（长期建设，提升工程质量）

7. **将脚本拆分为模块化函数文件**
   - `network.sh` / `wg.sh` / `cloudflare.sh` / `ui.sh`
   - 价值：后续迭代速度更快，减少改崩概率

8. **加入最小化回归测试（shellcheck + bats）**
   - 至少覆盖：配置读写、参数组装、落地切换流程
   - 价值：每次改动可自动发现明显问题

9. **systemd 模式（可选）**
   - 提供 screen 与 systemd 双运行模式
   - 价值：生产环境更稳、日志接入更标准
